MiFID II ‚Äî Multi-Field Trade Validation

Identifies reportable trades with data quality failures across the three fields most commonly rejected by trade repositories: ISIN, LEI, and execution timestamp. Runs against the prior trading day before the submission window opens. Each failure category is labelled so the breaks report can be triaged by type rather than reviewed manually line by line.

mifid2_validation.sql 
-- MiFID II pre-submission validation
-- Flags ISIN, LEI and timestamp failures before the 23:59 cutoff
-- Run against prior trading day: CURRENT_DATE - 1

SELECT
    t.trade_id,
    t.execution_timestamp,
    t.instrument_isin,
    t.executing_entity_lei,
    t.transaction_price,
    CASE
        WHEN t.instrument_isin IS NULL
            THEN 'FAIL_ISIN_NULL'
        WHEN t.instrument_isin !~ '^[A-Z]{2}[A-Z0-9]{9}[0-9]$'
            THEN 'FAIL_ISIN_FORMAT'
        WHEN t.executing_entity_lei IS NULL
            THEN 'FAIL_LEI_NULL'
        WHEN t.executing_entity_lei !~ '^[A-Z0-9]{18}[0-9]{2}$'
            THEN 'FAIL_LEI_FORMAT'
        WHEN t.execution_timestamp > NOW()
            THEN 'FAIL_TIMESTAMP_FUTURE'
        WHEN t.execution_timestamp < NOW() - INTERVAL '2 days'
            THEN 'FAIL_TIMESTAMP_STALE'
        ELSE 'PASS'
    END                          AS validation_status,
    t.reportable_flag,
    t.trade_date
FROM  trades t
WHERE t.reportable_flag = TRUE
  AND   t.trade_date    = CURRENT_DATE - 1
ORDER BY
    validation_status,
    t.execution_timestamp;

EMIR Refit ‚Äî Counterparty Data Reconciliation

Reconciles counterparty identifiers between the internal trade booking system and the trade repository acknowledgement file. Surfaces mismatches in LEI, notional, and UTI fields that would cause rejection on the next submission cycle.

emir_refit_recon.sql 
-- EMIR Refit counterparty reconciliation
-- Compares internal book vs trade repository ACK
-- Surfaces LEI, notional and UTI mismatches

SELECT
    b.trade_ref,
    b.counterparty_lei        AS book_lei,
    r.counterparty_lei        AS repo_lei,
    b.notional_amount         AS book_notional,
    r.notional_amount         AS repo_notional,
    b.uti                     AS book_uti,
    r.uti                     AS repo_uti,
    CASE
        WHEN b.counterparty_lei <> r.counterparty_lei
            THEN 'LEI_MISMATCH'
        WHEN ABS(b.notional_amount - r.notional_amount) > 0.01
            THEN 'NOTIONAL_MISMATCH'
        WHEN b.uti <> r.uti
            THEN 'UTI_MISMATCH'
        ELSE 'MATCHED'
    END                         AS recon_status
FROM       trade_book     b
INNER JOIN repo_ack_file  r
        ON b.trade_ref  = r.trade_ref
WHERE  b.report_date = TRUNC(SYSDATE) - 1
  AND  b.regime      = 'EMIR_REFIT'
ORDER BY recon_status, b.trade_ref;

SFTR ‚Äî Automated Exception Report

Reads the daily SFTR submission file, validates collateral type codes and settlement dates against the ESMA schema, and writes a structured exception report to CSV. Replaces a manual review step that previously took 45 minutes each morning.

sftr_exceptions.py 
import pandas as pd
from datetime import date, timedelta

# ESMA-permitted collateral type codes (SFTR Annex II)
VALID_COLLATERAL = {'GOVS', 'CORP', 'EQUI', 'CCPS', 'OTHR'}
REPORT_DATE = date.today() - timedelta(days=1)

def validate_sftr_file(filepath: str) -> pd.DataFrame:
    """
    Load SFTR submission file and return rows
    that fail collateral type or settlement date checks.
    """
    df = pd.read_csv(filepath, parse_dates=['settlement_date'])
    df = df[df['report_date'] == str(REPORT_DATE)]
    exceptions = []
    for _, row in df.iterrows():
        reason = None
        if row['collateral_type'] not in VALID_COLLATERAL:
            reason = f"Invalid collateral type: {row['collateral_type']}"
        elif pd.isnull(row['settlement_date']):
            reason = "Settlement date missing"
        elif row['settlement_date'].date() > REPORT_DATE:
            reason = "Settlement date in future"
        if reason:
            exceptions.append({
                'trade_id'       : row['trade_id'],
                'collateral_type' : row['collateral_type'],
                'settlement_date' : row['settlement_date'],
                'failure_reason'  : reason,
            })
    return pd.DataFrame(exceptions)

# Run and export
report = validate_sftr_file('sftr_daily.csv')
report.to_csv(f'sftr_exceptions_{REPORT_DATE}.csv', index=False)
print(f"{len(report)} exceptions written to CSV")

Pass Rate % =
VAR TotalRecords =
    CALCULATE(
        COUNTROWS( ValidationResults ),
        ValidationResults[validation_status] <> "EXCLUDED"

    CALCULATE(

CFTC ‚Äî Daily Submission Summary by Action Type

Produces the end-of-day summary used by compliance to sign off CFTC submissions. Breaks counts by action type (NEW, MODIFY, CANCEL, ERROR) so that the ratio of corrections to new reports is visible ‚Äî a spike in MODIFY or CANCEL rates typically signals an upstream data issue.

cftc_daily_summary.sql 
-- CFTC daily submission summary
-- Grouped by action type for compliance sign-off
-- Flag raises if MODIFY + CANCEL > 5% of total

SELECT
    action_type,
    COUNT(*)                              AS record_count,
    ROUND(
        COUNT(*) * 100.0
        / SUM(COUNT(*)) OVER (),
        2
    )                                     AS pct_of_total,
    CASE
        WHEN action_type IN ('MODIFY', 'CANCEL')
         AND COUNT(*) * 100.0
             / SUM(COUNT(*)) OVER () > 5
            THEN 'REVIEW'
        ELSE 'OK'
    END                                   AS flag
FROM    cftc_submissions
WHERE   submission_date = CURRENT_DATE - 1
GROUP BY action_type
ORDER BY record_count DESC;

Regime-Agnostic Break Classifier

Classifies validation breaks by severity and routes them to the correct remediation queue. Works across MiFID II, EMIR, SFTR, and CFTC by reading a config file that maps each regime's field rules. Critical breaks block submission; warnings are logged for next-day review.

break_classifier.py 
from dataclasses import dataclass
from enum       import Enum

class Severity(Enum):
    CRITICAL = "CRITICAL"  # blocks submission
    WARNING  = "WARNING"   # logged, next-day review
    INFO     = "INFO"      # informational only

@dataclass
class Break:
    trade_id : str
    field    : str
    reason   : str
    severity : Severity
    regime   : str

def classify_breaks(records: list, regime: str) -> list[Break]:
    """
    Classify a list of validation failures by severity.
    regime: 'MIFID2' | 'EMIR_REFIT' | 'SFTR' | 'CFTC'
    """
    CRITICAL_FIELDS = {
        'MIFID2'     : {'isin', 'lei', 'execution_timestamp'},
        'EMIR_REFIT' : {'uti', 'counterparty_lei', 'notional'},
        'SFTR'       : {'collateral_isin', 'settlement_date'},
        'CFTC'       : {'usi', 'clearing_indicator'},
    }
    critical = CRITICAL_FIELDS.get(regime, set())
    return [
        Break(
            trade_id = r['trade_id'],
            field    = r['field'],
            reason   = r['reason'],
            severity = Severity.CRITICAL if r['field'] in critical
                       else Severity.WARNING,
            regime   = regime,
        )
        for r in records
    ]
